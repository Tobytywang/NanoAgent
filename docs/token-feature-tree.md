# NanoAgent Token 特性树

> 按 Token 效率优化和 Token 准确性两条主线，以树形结构整理已完成和未完成的 feature。
>
> ✅ = 已实现　⏳ = 未实现（有规划）　🔴 = 已实现但存在准确性缺陷
>
> 标注规则：
> - 【已完成，版本：vX.X.X】— 已实现
> - 【待完成，计划：vX.X.X】— 已规划待实现
> - 【不建议，原因：xxx】— 列出但不建议实现

---

## 0. 可用性阈值分层

> 按设计重要性划分三个阈值：没有它系统不可用 → 没有它不够好 → 完整设计
>
> 划分原则：纯设计视角，不考虑实现难度或完成状态

### 0.1 最基本 — 生存条件（7 个）

> 没有它**任何场景都不可用**——系统会在某个不可预测的时刻失败

| # | Feature | 位置 | 线 | 为什么 |
|---|---------|------|------|--------|
| 1 | 字符计数启发式估算 | 2.1.1 | 准确性 | 首轮的唯一决策依据，没有估算函数系统连"该不该压缩"都无法判断 |
| 2 | 模型上下文窗口准确性 | 2.6.1 | 准确性 ✅ v0.7.11 | 所有百分比决策的分母，窗口错了压缩阈值和预算全部算错 |
| 3 | 三层上下文压缩 | 1.5.2 | 效率 | 多轮对话生存机制，没有压缩超过 3-4 轮必然溢出 |
| 4 | 工具结果智能摘要 | 1.2.3 | 效率 | 工具结果进入推理链的唯一过滤器，大文件原样灌入一次撑爆 |
| 5 | 多维预算约束 | 1.6.1 | 效率 | 防止病态行为的第一道闸，只控 token 不控迭代会死循环耗尽 API 配额 |
| 6 | 预算管理（追踪+强制总结） | 1.6.2 | 效率 | 预算耗尽无输出用户白等，"有结果返回"的基本保证 |
